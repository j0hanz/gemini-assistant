# Refactor task error surface to preserve input-validation messages

When a client calls a public tool (`chat`, `research`, `analyze`, `review`, `memory`) with task augmentation (`params.task` present), any throw inside the inner `tools/call` dispatch path — most commonly client-supplied input-validation failures — is rewritten by `@modelcontextprotocol/server` (alpha) into the opaque JSON-RPC `-32602` error:

```
Invalid task creation result: [{ path: ["task"], expected: "object", received: "undefined" }]
```

This plan makes task-augmented calls never throw at the transport boundary. Instead, failures are materialized as a valid `CreateTaskResult` whose associated stored task result carries the real error message. Clients then retrieve the true failure via `tasks/result`, matching the documented task lifecycle and eliminating the misleading alias error.

## 1. Requirements & Constraints

- **REQ-001**: Every task-augmented `tools/call` for the five public tools MUST return a schema-valid `CreateTaskResult` (`{ task: { taskId, status, ttl, createdAt, lastUpdatedAt, ... } }`) to the client, even when input validation, handler setup, or business logic fails.
- **REQ-002**: When the underlying failure is an input-validation error, the created task MUST reach status `failed` and its stored result MUST be `{ content: [{ type: 'text', text: <real error message> }], isError: true }`.
- **REQ-003**: Non-task `tools/call` behavior MUST be unchanged: invalid input still maps to a standard tool error result (`isError: true`) via the existing SDK path.
- **REQ-004**: The error message preserved in `tasks/result` MUST include the SDK’s validation message verbatim (e.g. `Input validation error: Invalid arguments for tool <name>: <zod-error>`), so clients and log scrapers can diagnose without server-side logs.
- **REQ-005**: `taskSupport: 'optional'` automatic polling (sync-style `tools/call` without `task` param, but tool is task-based) MUST continue to work and surface failures as normal `CallToolResult.isError = true`.
- **SEC-001**: Error text written into `storeTaskResult` MUST NOT include internal stack traces or secrets; only the `Error.message` string, matching existing `AppError.toToolResult()` behavior.
- **CON-001**: No fork or patch of `@modelcontextprotocol/server` source may be applied. The fix must live in this repository and survive SDK dependency updates within the same alpha major.
- **CON-002**: The existing public schema contracts in [src/schemas/outputs.ts](C:/gemini-assistant/src/schemas/outputs.ts) and [src/public-contract.ts](C:/gemini-assistant/src/public-contract.ts) MUST NOT change.
- **CON-003**: `npm run format`, `npm run lint`, `npm run type-check`, and `npm run test` MUST pass after the change.
- **GUD-001**: Keep the fix centralized in [src/lib/task-utils.ts](C:/gemini-assistant/src/lib/task-utils.ts) so no per-tool registration site needs modification.
- **GUD-002**: Log the original failure once, at `error` level, through the existing `logger.child('task-utils')` scope, with `toolName`, `taskId`, and sanitized `args` summary (`maybeSummarizePayload`). Do not duplicate logging.
- **PAT-001**: Follow the existing `executor.run` → `runToolAsTask` pattern: create the task first, then run work inside the task; on failure, call `store.storeTaskResult(taskId, 'failed', { content, isError: true })` and never rethrow from the `createTask` handler.
- **PAT-002**: Use `AppError.from(err, toolName).message` to normalize error text, mirroring the convention used in [src/lib/tool-executor.ts](C:/gemini-assistant/src/lib/tool-executor.ts).

## 3. Alternatives

- **ALT-001**: Patch `@modelcontextprotocol/server` via a `patch-package` postinstall step to remove the outer `CreateTaskResultSchema` wrapper. Rejected per CON-001: fragile across SDK updates and hides behavior the SDK will eventually stabilize.
- **ALT-002**: Re-register the `tools/call` handler in `src/server.ts` after `McpServer` setup and fully replace the inner dispatch. Rejected: duplicates SDK logic for capability checks, `handleAutomaticTaskPolling`, and `tasks/list` wiring; high maintenance surface.
- **ALT-003**: Do nothing and document the masked error in the README. Rejected per REQ-004: the current behavior actively misleads clients and masks legitimate validation bugs, which is the motivating complaint.
- **ALT-004**: Move input validation entirely out of the SDK by passing `inputSchema: undefined` at registration time and performing all validation inside `wrapTaskSafeWork`. Considered; downgraded to TASK-012 fallback because it also removes `tools/list` schema advertisement unless re-added separately.

## 7. Risks & Assumptions

- **RISK-001**: Removing `inputSchema` from SDK registration (TASK-011 / TASK-012 fallback path) could silently change `tools/list` advertisement. Mitigation: keep `inputSchema` on registration and rely solely on Phase 2’s `storeTaskResult` fallback; only pursue TASK-011 if Phase 2 proves insufficient against the SDK’s inner `try/catch`.
- **RISK-002**: `McpServer` internal behavior may change in future `@modelcontextprotocol/server` alphas (e.g., the outer `CreateTaskResultSchema` wrapper might be removed). Mitigation: keep the fix local to `task-utils.ts` so it can be deleted in one place if the SDK stabilizes.
- **RISK-003**: `runToolAsTask` already stores `'failed'` for rejected promises. The new `wrapTaskSafeWork` layer must not double-store results. Mitigation: `wrapTaskSafeWork` ALWAYS resolves with a `CallToolResult` (never throws), and `runToolAsTask` branches on `result.isError` for the `'failed'` status — single write per outcome.
- **RISK-004**: Logging task-failure details may include user-supplied content. Mitigation: reuse `maybeSummarizePayload` from [src/lib/logger.ts](C:/gemini-assistant/src/lib/logger.ts) (GUD-002) which honors `verbosePayloads` configuration.
- **ASSUMPTION-001**: The SDK’s inner `tools/call` handler catches `ProtocolError(InvalidParams)` from `validateToolInput` and converts it into `createToolError(...)`, producing the shape that fails `CreateTaskResultSchema`. Verified by reading `node_modules/@modelcontextprotocol/server/dist/index.mjs` lines 985–1005 during investigation on 2026-04-21.
- **ASSUMPTION-002**: `InMemoryTaskStore.storeTaskResult` never throws for a freshly-created task ID with a terminal status transition. Backed by SDK behavior; no explicit contract guarantee.
- **ASSUMPTION-003**: All five public tools register via `registerTaskTool` (verified in `src/tools/{chat,research,analyze,review,memory}.ts`), so centralizing the fix in `task-utils.ts` covers 100% of user-facing task paths.
